Each implemented write operation must have at least one success case, one validation failure, one authorization failure when a denied fixture exists, one rollback/no-partial-mutation case, one read-after-write check, and final cleanup proof. If a denied fixture does not exist, record the permission row as `Deferred` with the exact fixture gap and do not claim proven `403` parity.

For delete/update operations, include dependency cases for both direct association blockers and transitive rows generated by triggers/packages. Record the setup route and cleanup route for every controlled fixture. Prefer disposable data; when an existing development record is the necessary parity fixture, capture the sanitized baseline and prove restoration. In a workspace with `development-mutation-authority.md`, do not request individual approval for that controlled mutation.

For negative probes, include before/after counts for the target table and known side-effect tables. If the legacy call unexpectedly creates data, cleanup must use the chosen legacy route and the matrix must link the final zero-count proof.

Do not mark permission parity closed from UI button state. Use a real user/profile fixture and capture the routine's behavior. Create permission and delete permission may be separate cases.

After API implementation, add API smoke state to each row:

- `Smoke closed` only when request/response, DB side effects, readback when applicable, and cleanup proof are all recorded.
- `Deferred` only when the matrix names the missing fixture/evidence and explains why it is outside the current pilot.
- `Backend implemented` is not enough for parity closure.

For a pilot release, add a `Pilot Deferrals` section listing every unsupported operation or parity gap, such as delete permission, dependency-blocked delete, pending behavior, publication/legal payloads, update, duplicate button behavior, and cancel/no-op.

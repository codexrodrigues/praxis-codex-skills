# Praxis Angular i18n Governance

Use this skill for framework-owned text in `@praxisui/*`. Internal UI and authoring chrome are platform text; domain labels and business content remain owned by host metadata/config/schema.

## Canonical Rule

Do not introduce visible hardcoded text in components, directives, UX services, dialogs, snackbars, tooltips, `aria-label`, placeholders, hints, validation messages, empty states, or default labels.

Use the lib's existing i18n path or the shared localization layer in `@praxisui/core`. The canonical authoring pattern is `PraxisI18nService` plus a lib namespace/catalog registered through the appropriate provider.

For shared core i18n, resource-discovery copy, value presentation, translator providers, or cross-lib framework text, use `praxis-core-i18n-resource-copy` before changing catalogs or copy.

## Required Inventory

Before editing:

1. Read `praxis-ui-angular/AGENTS.md` and the local `projects/<lib>/AGENTS.md` when present. If local AGENTS is absent, use `praxis-angular-agents-governance` and derive i18n expectations from README, i18n catalogs, metadata, docs, focused specs, and root governance.
2. Inspect existing i18n files for the lib, commonly under `projects/<lib>/src/lib/i18n/**` or adjacent `*.i18n.ts`.
3. Check `tools/i18n/reports/catalog-inventory.md`, `catalog-coverage.md`, `hardcoded-text-hotspots.md`, and `locale-formatting-map.md` when doing audits or broader cleanup.
4. Separate framework chrome from domain text:
   - framework chrome: translate through Praxis i18n
   - domain/business labels: preserve schema/config/metadata values from the host
5. If the change touches an authoring editor, use `praxis-authoring-editors` as the contract owner for round-trip parity.

## Implementation Guidance

- New internal text should have translation keys and controlled fallbacks.
- Update at least `pt-BR` and `en-US` catalogs when the lib has locale files.
- Brazilian Portuguese files must use `.pt-BR.ts`, not `.pt.ts`.
- Preserve UTF-8 and correct PT-BR accents and grammar.
- Prefer established Portuguese technical terms, such as "Gráfico" rather than "Chart", except when the English term is the technical platform name.
- Do not expose raw `err.message` to end users without UX normalization and i18n.
- If a public API still accepts `string`, keep compatibility but route internal defaults through i18n where possible.

For authoring editors, catalog labels, sections, tab names, placeholders, helper text, empty states, defaults generated by helper schemas, action specs, descriptors, or form defaults. JSON-only support for user-facing text is incomplete when the platform owns the editor chrome.

## Dynamic Fields And Authoring

For `@praxisui/dynamic-fields`, do not look only for component templates. Also inspect:

- component registry metadata
- editorial descriptors
- field catalogs
- metadata-editor consumption
- inline filter/tooling coverage

For Settings Panel, Page Builder, table, form, charts, and visual builders, update the shared catalog of the owning lib rather than scattering microcatalogs for each component unless that is already the local convention.

For `@praxisui/cron-builder`, separate internal builder text, validation messages, preset labels, preview labels, and humanized schedule text from host domain schedule names, owners, tags, and governance descriptions. Use `praxis-cron-builder-runtime` or `praxis-cron-schedule-authoring` for the runtime/source boundary before changing locale behavior.

For `@praxisui/table-rule-builder`, keep panel tabs, preset names, placeholders, aria labels, advanced field labels, preview copy, and validation messages in the `praxisTableRuleBuilder` i18n namespace. Use `labels` only as host override above the catalog and never persist it into `RuleEffectDefinition`.

## Validation

Choose the smallest reliable validation:

- focused specs for the changed component/service/catalog if present
- build/test for the owning lib when catalog exports or providers changed
- browser/screenshot QA when the visible authoring surface changed
- hardcoded text audit or report review for broader cleanup

Use `praxis-angular-validation-gates` for command selection. Use `praxis-angular-docs-playgrounds` when public docs, examples, or playground text must be synchronized.

## Output Expectations

Report:

- which text is framework chrome vs domain content
- namespace/catalog updated
- `pt-BR` and `en-US` coverage
- validation run
- any remaining hardcoded text intentionally left and why

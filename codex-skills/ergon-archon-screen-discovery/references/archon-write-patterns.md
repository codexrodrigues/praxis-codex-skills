# Archon Write Patterns

Use this reference when a screen exposes create, edit, delete, duplicate, publication, legal-document, or pending-record behavior. The goal is to avoid modeling legacy writes as direct CRUD before the Archon/Ergon write stack is understood.

## Architecture Lesson

Archon screens often look like simple table maintenance, but writes commonly pass through generated DML routines, table triggers, audit packages, publication/legal-document components, and pending-record workflows.

Treat this as the default hypothesis, not as proof for every screen. Confirm the exact objects from XML/runtime evidence and Oracle metadata before designing a write API.

## Common Write Stack

1. Browser/XML identifies the visible write area, usually a `recordPanelEdit` or reusable block.
2. The component points to a `rotina`, often `db:<procedure>`.
3. For generated table maintenance, the routine may be an `ERG_DML_*` procedure generated by `HAD_GERA_PROC_DML`.
4. Generated DML routines commonly bind the selected row through rowid and receive domain fields, operation flags, output messages, and sometimes publication parameters.
5. Target tables may have generated triggers and table packages:
   - `T_BS_IUD_<TABELA>`
   - `T_B_IUD_<TABELA>`
   - `T_A_IUD_<TABELA>`
   - audit triggers such as `AUDIT_*` / `AUDHD_*`
   - `PCK_<TABELA>`
6. Triggers/packages may depend on `HADES.FLAG_PACK` or related package state for user, company, role, transaction, machine, and external-user context.
7. Pending flow may redirect real insert/update/delete into `<TABELA>_PND`, `PCK_<TABELA>_PND`, `PACK_HAD_PEND`, and `PACK_ERG_PEND`.
8. Publication and legal-document blocks may add `PONTPUBL`, `PONTLEI`, rowid parameters, side tables, commits, and separate routines.

## What To Investigate In XML

For every write-capable component, record:

- component id, visible tab/section, and component type;
- `rotina`, especially `db:*` entries;
- `paramId`, selected-row parameters, `ROWID`, `ID_REG`, and hidden fields;
- field names and bind order passed to the routine;
- reusable block references such as `HADadm_blk001` and `ERGadm_blk004`;
- links to `ERGadmREGPEND` or pending-detail screens;
- create/edit/delete permission hints from HADES access pattern tables;
- validation messages, popup flows, and enabled/disabled states observed in the browser.

## What To Investigate In Oracle

Start from exact routine and target table names found in XML/runtime evidence.

- `ALL_ARGUMENTS`: routine contract and `PP_%` publication parameters.
- `ALL_SOURCE`: routine source slices for `HAD_GERA_PROC_DML`, `dbms_sql.bind_variable_rowid`, validation code, errors, `FLAG_PACK`, `PACK_HAD_PEND`, `PACK_ERG_PEND`, and commits/autonomous transactions.
- `ALL_TRIGGERS` and `ALL_SOURCE`: generated triggers on the target tables.
- `ALL_OBJECTS`: `PCK_<TABELA>`, `<TABELA>_PND`, and `PCK_<TABELA>_PND`.
- `ALL_DEPENDENCIES`: packages/functions called from routines and triggers.
- `ALL_TAB_COLUMNS`: special columns such as `ID_REG`, `PONTPUBL`, `PONTLEI`, `DTINI`, `DTFIM`, `CHAVE_PEND`, `STATUS_PEND`, and audit fields.
- `ALL_TAB_PRIVS`: grants for routines, target tables, pending tables, and helper packages.

## API Migration Rule

Do not design a write endpoint as direct CRUD until this stack is checked.

For first-phase read migrations, it is acceptable to defer write APIs only when the deferral lists the exact routines, triggers, packages, audit/session context, pending flow, publication/legal coupling, and parity cases still required.

For write APIs, decide explicitly whether the new service will:

- call the legacy routine on the same initialized Oracle session;
- reproduce the legacy routine and trigger semantics in Java;
- keep writes blocked until the pending/publication/legal flows are fully modeled.

## ERGadm00189 Example

The frequency screen showed the pattern clearly:

- XML write component calls `db:erg_dml_freq_formato`.
- The write routine receives frequency fields plus publication fields.
- The flow validates quantity/code format.
- Writes touch `FREQUENCIAS` through generated DML and table triggers.
- Audit/session context and pending behavior are involved.
- `PCK_FREQUENCIAS_PND` can redirect or later apply pending changes.

Therefore ERGadm00189 is not a simple CRUD API over `FREQUENCIAS`; it is a useful baseline pattern for investigating similar maintenance screens.
